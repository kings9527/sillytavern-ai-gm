# 2026-06-10 问题备案记录

## 核心问题：Cron 任务因 Message 模糊导致超时

**时间：** 06-10 10:00（已修复）、13:00（修复后成功）、20:00（重新触发后成功）
**表现：** isolated session 运行 30 分钟后 timeout，0 产出，0 token，无交付内容

**根本原因：**
- 任务 message 包含模糊指令（如"根据上午进度继续 Phase 3"）
- isolated session 花全部时间分析状态（读文件、判断进度、决策），30 分钟内未写任何代码
- 对比：明确指令（"创建 X 文件，实现 Y 功能"）的 10:00 任务在 2 分钟内完成

**已解决：** 所有任务 message 已改为精确文件目标 + 具体函数名 + 技术细节

**预防措施：**
- 任务 message 必须明确：文件路径 + 函数名 + 输入输出 + 技术约束
- 禁止"继续 Phase 3"、"推进开发"等模糊描述
- 必须包含"每轮只改 1-2 个文件，逐个完成"

## 问题 2："只创建 1 个文件" vs "每轮 1-2 个文件" 歧义

**时间：** 06-10 20:52
**表现：** 用户要求"做多个文件，但一个一个做"，但 message 写的是"只创建 1 个文件"

**根本原因：**
- isolated session 可能理解为"整个任务只能做 1 个文件"
- 实际是"每轮做 1-2 个，分多轮完成"

**已解决：** 统一措辞为"每轮只改 1-2 个文件，逐个完成，产出 < 3k 行"

**预防措施：**
- 所有任务 message 统一使用标准措辞："每轮只改 1-2 个文件，逐个完成"
- 避免"只创建 X 个文件"（暗示总数限制）
- 使用"如果本轮完成且时间充裕，继续处理下一个文件"

## 问题 4：任务时间段内文件被重复/冲突修改

**时间：** 06-10 10:00-13:00 之间
**表现：** 10:00 任务创建的文件，13:00 任务以为未完成，试图重复创建

**根本原因：**
- 13:00 任务 message 模糊（"根据上午进度继续"），isolated session 不知道 10:00 已完成
- 没有任务间状态同步机制（除了 project-status.md，但读取耗时）

**已解决：** 改为精确文件目标 + 先检查再修改（"如果缺失则创建"）

**预防措施：**
- 任务 message 使用条件式："如果文件已存在，跳过；如果缺失，创建"
- 每个开发轮次开始前检查文件系统状态
- 使用 project-status.md 作为状态源，但减少读取次数（先 check 本地文件，再 check status）

## 明日遇到这些问题的处理预案

| 问题 | 出现征兆 | 处理措施 |
|------|----------|----------|
| 任务超时（30分钟0产出） | 日志显示 durationMs=1800000，deliveryStatus=unknown | 立即检查 message 措辞，改为精确文件目标，重新触发 |
